## Technical Choices
- Used Svelte 5 local state primitives (`$state`, `$derived`) rather than introducing a global store. The network page is currently prototype-local and does not yet persist to Convex.
- Kept the palette monochrome to match the chat surface and leave room for future brand color decisions. Colors are expressed in OKLCH with zero chroma, following the existing chat route’s black/gray/white styling direction.
- Implemented the Rolodex as a scrollable 3D stack using CSS `perspective`, `rotateX`, `translateZ`, and a bottom-origin transform. This makes cards feel like they rotate upward from a lower attachment point rather than hanging from a top spindle.
- Removed the visible spindle after it became visually dominant and obscured information. The physical metaphor now comes from motion and card geometry rather than a literal bar.
- Split the focused selected card out of the Rolodex stack and render it as a separate fixed-position card at the page level. This avoids clipping and transform-context bugs caused by fixed elements inside transformed/perspective scroll containers.
- Kept the background Rolodex visible but de-emphasized during focus with blur and opacity, so the user retains spatial context while reading the selected card.
- Added a transparent full-screen dismiss layer behind the focused card for click-off dismissal. The CSS had to explicitly exclude this layer from generic button styling to avoid the earlier “giant black pill” bug.
- Added scroll-to-focus behavior: clicking a visible region of the stack maps the click position to the nearest card slot, scrolls that card toward the apex if needed, then opens it.
- Added scroll-away dismissal: wheeling while a card is focused clears selection, then applies a reduced/resisted scroll delta to the Rolodex.

## Reasoning
The requested page is not a generic CRM table. The goal is a relationship-intelligence surface that feels like a tactile personal network object. A Rolodex interaction supports that product metaphor better than a grid of cards because it implies memory, browsing, recency, and intentional inspection. The focused card pattern lets the collapsed stack stay lightweight while still exposing schema-relevant details when the user asks for them.

The decision to render the selected card separately was important. CSS transforms and perspective create containing blocks that can make `position: fixed` behave as though it is fixed to the transformed ancestor instead of the browser viewport. Moving the focused card out to the page level repaired clipping and made browser-centered focus reliable.

## Learnings
- A transformed/perspective ancestor can break expectations around fixed positioning and viewport-centered overlays.
- Broad button styling can accidentally style invisible overlay controls; special-purpose overlay buttons should be excluded from generic button rules.
- For this product, a focused-card overlay gives a cleaner reading/editing mode than trying to expand cards in-place inside the scroll stack.
- The schema already suggests a useful front-end information hierarchy: person identity, current organization/membership, last interaction/event, and notes.
